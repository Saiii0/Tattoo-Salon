Документ: Архитектура и структура приложения "Tattoo Salon"
Дата: 2026-04-02

1) Суть проекта в одном абзаце
Это веб‑приложение для тату‑салона. Есть клиент (SPA на React), сервер (Express API) и база данных (PostgreSQL). Клиент рисует страницы и при любом действии ходит на сервер по HTTP. Сервер валидирует права, читает и пишет данные в базу, возвращает JSON. Авторизация через JWT, токен хранится в localStorage.

2) Общая схема, «кто откуда что подтягивает»
UI (React страницы) -> AppContext -> api.ts -> fetch('/api/...')
fetch -> Vite proxy -> Express сервер -> PostgreSQL
PostgreSQL -> Express сервер -> JSON ответ
JSON -> AppContext state -> UI ререндер

Файлы, которые держат эту цепочку:
- Клиент вызывает API: client/src/app/api.ts
- Клиент хранит состояние: client/src/app/context/AppContext.tsx
- Сервер обрабатывает API: server/src/index.js
- Сервер ходит в БД: server/src/db.js
- БД схема и сиды: server/sql/migrations/*.sql

3) Что где лежит (минимальная карта)
Корень проекта
- client/ — фронтенд (React + TypeScript + Vite)
- server/ — бэкенд (Express + PostgreSQL)
- ARCHITECTURE.txt — этот документ

4) Клиент (front)
4.1 Точка входа
Файл client/src/main.tsx
- Поднимает React приложение
- Подключает глобальные стили

4.2 Роуты и защита
Файл client/src/app/routes.tsx
- /login, /register — публичные
- Внутренние страницы оборачиваются в ProtectedRoute
- Для admin/master/user есть отдельные ограничения

Файл client/src/app/components/ProtectedRoute.tsx
- Если не авторизован: показывает Unauthorized
- Если роль не подходит: показывает Forbidden

4.3 Глобальное состояние
Файл client/src/app/context/AppContext.tsx
- Держит currentUser, users, services, orders, reviews, favorites
- Инициализирует данные через API при старте
- После login/register сохраняет токен и загружает данные

4.4 Как клиент ходит на сервер
Файл client/src/app/api.ts
- Базовый url: /api
- Если есть токен, кладет Authorization: Bearer <token>
- Возвращает JSON или кидает ошибку

4.5 Проксирование запросов
Файл client/vite.config.ts
- /api и /uploads идут на http://localhost:3001
- Поэтому в коде можно писать просто /api/..., не указывая хост

4.6 Типы данных
Файл client/src/app/types.ts
- User, Service, Order, Review и прочие типы
- Вся логика в клиентах опирается на эти типы

4.7 Важно
Файл client/src/app/data/mockData.ts есть, но сейчас НЕ используется.
Данные берутся с сервера, а не из моков.

5) Сервер (back)
5.1 Точка входа
Файл server/src/index.js
- Запускает Express
- Подключает CORS, JSON парсинг
- Настраивает JWT
- Подключает Swagger UI на /api/docs
- Раздает статические файлы /uploads

5.2 Подключение к базе
Файл server/src/db.js
- Использует переменные окружения из server/.env
- Подключается к PostgreSQL

5.3 Миграции и сиды
Файл server/src/migrate.js
- Создает БД (если нет)
- Применяет SQL из server/sql/migrations

5.4 Файлы миграций
server/sql/migrations/001_schema.sql
- Таблицы users, services, orders, reviews, favorites, client_history

server/sql/migrations/002_seed.sql
- Начальные пользователи и данные

server/sql/migrations/004_review_reactions.sql
- Таблица реакций на отзывы (лайк/дизлайк)

6) Основные API эндпоинты
Серверный файл: server/src/index.js

Аутентификация
- POST /api/auth/register
- POST /api/auth/login
- GET /api/auth/me

Пользователи
- GET /api/users (admin)
- POST /api/users (admin)
- PATCH /api/users/:id
- DELETE /api/users/:id (admin)

Услуги
- GET /api/services
- POST /api/services (admin)
- PATCH /api/services/:id (admin)
- DELETE /api/services/:id (admin)
- GET /api/services/:id/reviews
- GET /api/services/:id/history

Отзывы
- GET /api/reviews
- POST /api/reviews (user)
- PATCH /api/reviews/:id/like
- PATCH /api/reviews/:id/dislike
- DELETE /api/reviews/:id (admin)

Заказы
- GET /api/orders
- POST /api/orders (user)
- PATCH /api/orders/:id (user)
- PATCH /api/orders/:id/status (master/admin)

Избранное
- GET /api/favorites (user)
- POST /api/favorites/:serviceId (user)

Расписание

7) Как работает авторизация (простыми словами)
1. Пользователь логинится на /api/auth/login
2. Сервер проверяет пароль и выдает JWT
3. Клиент кладет JWT в localStorage под ключом access_token
4. При каждом запросе клиент добавляет заголовок Authorization: Bearer <token>
5. Сервер проверяет токен и решает, пускать или нет

8) Как работает загрузка аватара
1. Клиент вызывает usersApi.uploadAvatar
2. Отправляется FormData с файлом avatar
3. Сервер сохраняет файл в server/uploads
4. Сервер возвращает путь /uploads/имя_файла
5. Этот путь сохраняется в users.avatar
6. Клиент отображает картинку по /uploads/...

9) Как работает расписание
1. Клиент вызывает /api/schedule с date + serviceId + masterId
2. Сервер берет длительность услуги
3. Сервер собирает занятость мастера по заказам
4. Сервер строит тайм‑слоты с шагом 30 минут
5. Сервер возвращает slots и busy

10) Как работают страницы (логика на пальцах)
Home
- Показывает услуги из services

ServiceDetail
- Получает details по услуге и отзывы
- Дает добавить в избранное
- Дает записаться (создает order)

Orders (master)
- Показывает заказы мастера
- Дает менять статус

MyOrders (user)
- Показывает заказы пользователя
- Дает отменить pending заказ

ManageServices (admin)
- CRUD по услугам

UserManagement (admin)
- CRUD по пользователям

12) Важные технические факты
- Токен живет в localStorage под ключом access_token
- Сервер использует JWT и проверяет роль через middleware
- Пароли хранятся в БД как хэш (bcrypt)
- Если сиды были с plain‑text, при первом логине пароль хешируется
- API документация доступна на /api/docs
